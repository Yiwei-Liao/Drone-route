# 假设与边界说明

本文档用于记录项目中的假设、不可验证信息、技术边界和后续需要补充的数据。所有无法从现有数据直接验证的内容，均不得作为确定结论输出。

## 一、当前可验证的信息

基于现有输入文件，可以验证：

1. 巡检任务统计表中有哪些任务字段。
2. 线路台账中有哪些杆塔坐标和线路信息。
3. KML 文件中是否包含航点、航点高度和扩展任务字段。
4. 每条 KML 航线的航点数量、航线长度、高度范围和航点间距。
5. KML 航线与线路台账中杆塔坐标之间的空间距离关系。
6. 基于已有字段的简单统计指标，例如飞行距离、飞行时长、电池余量、风速等。

## 二、当前不可直接验证的信息

当前不能直接验证：

1. KML 是否完整覆盖整条线路。
2. KML 是否代表实际飞行轨迹，还是预规划航线。
3. KML 航点高度是绝对高度、相对起飞点高度，还是相对地面高度。
4. 巡检任务统计表中的“飞行距离”单位和计算口径，除非原始字段有明确说明。
5. 杆塔连接顺序是否等于 Excel 行顺序。
6. 线路是否存在分支、同名线路、多回线路或特殊编号规则。
7. 电池消耗与气象、负载、爬升高度之间的严格物理关系。
8. 基站通信覆盖和真实链路质量。
9. 地形遮挡风险。
10. 建筑物、树木、山体、道路、河流、禁飞区等空间环境影响。

## 三、关于 KML 的假设

当前 KML 文件更可能是单塔或局部巡检航线，而不是完整长距离巡线航迹。

因此：

1. 不应直接用少量 KML 推断整个区域巡检规律。
2. 不应默认 KML 文件代表从机场到线路再返航的完整飞行过程。
3. 不应默认 KML 中所有扩展字段都能被通用 KML 解析器读取。
4. 需要单独解析 DJI/MIS 扩展字段，例如速度、航向、云台角度、转弯模式等。

## 四、关于地形数据的边界

当前没有真实 DEM/DSM 地形数据，因此：

1. 不得输出真实地形高程。
2. 不得输出真实 AGL 离地高度。
3. 不得输出真实地形遮挡结论。
4. 不得伪造山体、坡度、地表起伏或建筑物信息。

后续如需地形增强，应接入 DEM/DSM 数据，例如 GeoTIFF、HGT、ASC 等格式。Google Earth 可用于人工校核或展示参考，但不应作为可自由提取的地形计算数据源。

## 五、关于通信分析的边界

当前表格中即使存在基站经纬度，也不足以完成严格通信覆盖分析。

严格通信分析还需要：

1. 频段。
2. 带宽。
3. 发射功率。
4. 接收灵敏度。
5. 天线增益。
6. 天线高度。
7. 方位角。
8. 下倾角。
9. 地形遮挡。
10. 建筑物遮挡。
11. 无人机天线参数。

当前阶段最多只能计算：

1. 航点到基站的直线距离。
2. 航点是否超过设定距离阈值。
3. 是否需要进一步视距分析。

## 六、关于能耗模型的边界

当前可基于飞行距离、飞行时长、电池余量做经验统计，但不能直接建立严格物理能耗模型。

严格能耗模型还需要：

1. 无人机型号参数。
2. 电池容量。
3. 飞行速度曲线。
4. 载荷重量。
5. 风速和风向。
6. 爬升/下降功率。
7. 返航策略。
8. 悬停时间。
9. 云台或传感器工作负载。

当前阶段只建议做经验型风险评分，例如：

1. 电池余量偏低。
2. 航程明显过长。
3. 飞行时长异常。
4. 风速偏高。
5. 距离机场过远。

## 七、关于风险评分的边界

当前风险评分是规则模型，不是经验证的安全评估模型。

风险评分只能用于：

1. 数据探索。
2. 原型展示。
3. 辅助识别异常任务。
4. 后续专家规则或模型训练的基础。

不能用于：

1. 直接指导真实飞行作业。
2. 替代现场安全评估。
3. 替代无人机飞控系统。
4. 替代电力巡检规程。

## 八、后续需要补充的数据

为了从原型升级为较完整仿真系统，建议补充：

1. 更多 KML 航线文件。
2. 明确 KML 文件与任务统计表之间的对应关系。
3. DEM/DSM 地形数据。
4. 杆塔连接顺序或线路拓扑。
5. 无人机型号参数。
6. 电池容量与性能曲线。
7. 通信基站参数。
8. 机场/机巢准确坐标。
9. 禁飞区、道路、河流、建筑物等环境数据。
10. 巡检任务的人工标注结果或异常记录。

## 九、输出结论的标注要求

所有报告和界面输出应区分：

1. 可验证事实：直接来自原始数据或确定计算。
2. 假设：基于当前数据做出的前提设定。
3. 推断：由规则或空间关系推导出的结论。
4. 待验证：需要人工或补充数据确认的内容。
5. 无确切信息：当前数据无法支持的内容。

## 十、阶段 2 数据管线当前记录

阶段 2 已生成 `data/processed/` 标准 CSV/GeoJSON 和 `output/data_pipeline_log.json`。当前日志中的重要边界如下：

1. 规范目录 `data/raw/tasks`、`data/raw/ledger`、`data/raw/kml` 已预留，但原始文件仍位于 `raw/`；管线只读回退扫描 `raw/`。
2. 线路台账中 `35kV金茅线` 等部分杆塔缺少有效经纬度，已在日志中逐行记录。
3. 任务表两个时间字段原始表头重复，第二个字段暂记为 `end_time_raw_pending_confirm`，含义待确认。
4. 任务表“无人机飞行巡检距离（mk）”单位仍待确认，标准字段单位标记为 `unknown`。
5. 台账未提供机场/机巢经纬度，`airports.csv` 只保存名称，不生成伪造点位。
6. `lines.geojson` 当前按台账行顺序连接杆塔，仅作为线路候选几何；真实线路拓扑和分支关系需要人工或补充数据确认。

## 十一、阶段 3/4 接口边界

1. Web/GIS 当前版使用 React + CesiumJS 展示三维低空数字沙盘；这只是可视化原型，不是完整三维数字孪生。
2. `backend/app.py` 中的 `/api/risk` 为规则型原型风险评分，只用于数据探索，不可作为真实飞行安全结论。
3. `/api/terrain/sample` 调用 `TerrainSampler`，当前无 DEM/DSM 时返回 `DEM not provided`。
4. `/api/communication/check` 只计算航点到基站的 WGS84 直线距离，并提示是否超过阈值和是否需要视距分析；不输出覆盖强度、信号质量或遮挡结论。

## 十二、Cesium 三维可视化边界

1. Cesium 三维场景中的杆塔高度优先使用台账 `tower_height`；缺失时使用默认展示高度 `35 m`，并在属性中标注 `default height`。
2. KML 航线高度优先使用 KML altitude；缺失时使用默认展示高度 `80 m`，并标注 `simulated/default height`。
3. 当前 KML 没有真实时间戳，因此前端不提供无人机飞行动画回放；航点序列只用于航线展示、属性查看和通信链路取样。
4. 当前未接入真实 DEM/DSM，因此 Cesium 中的高度不等于真实离地高度 AGL。
5. Google Photorealistic 3D Tiles 仅作为可选展示底图预留，API key 从 `.env` 读取；没有 key 时自动使用基础三维场景。
6. 不允许从 Google 3D Tiles 提取、缓存或反向解析地形、建筑或高程数据。
7. 当前三维场景展示的是任务表中的基站点；台账中的机场/机巢只有名称，缺少真实经纬度，因此不生成伪造机场点。

## 十三、无底图 fallback 的边界

当前三维页面在无 Google/Cesium token 或在线瓦片失败时，会显示本地网格和数据叠加层。该 fallback 的边界如下：

1. 网格底图是本地渲染的显示背景，不是真实影像。
2. 数据叠加层使用真实经纬度进行平面投影，但不是 DEM/DSM，也不是地形模型。
3. 叠加层中的杆塔、航线、航点和基站候选点来自 `data/processed/` 标准化数据。
4. 叠加层不输出真实海拔、离地高度、通信覆盖或飞行安全结论。
5. Google 3D Tiles 仅作为可选显示底图，不允许提取、缓存或反向解析地形/建筑/高程数据。

## 十四、Cesium World Terrain 的边界

1. `frontend/.env` 中配置 `VITE_CESIUM_ION_TOKEN` 后，前端可加载 Cesium World Terrain 作为三维地形展示。
2. Cesium World Terrain 不是本项目的工程高程计算源，不用于输出真实海拔、AGL、净空或飞行安全裕度。
3. 当前不从 Cesium World Terrain 提取、缓存或反向解析地形数据。
4. 杆塔、航线、航点在三维展示中使用 `display_height_offset_m` 可视化偏移，目的是避免真实地形起伏遮挡图层；该偏移不是实际离地高度。
5. 无 token、token 无效或网络请求失败时，页面会回退到 fallback basic scene 或 Ellipsoid terrain，并保留业务图层展示。

## 十四-A、通信遮挡半透明图层的边界

当前前端提供 `通信遮挡演示` 图层，用半透明 Fresnel 走廊、链路中心线、所需覆盖半径圆和航线风险色带展示山体对无人机通信的可能影响。该图层的边界如下：

1. 这是视觉原型，不是实测通信覆盖、链路预算或飞行安全结论。
2. 通信源自动取距离当前选中 KML 航线起点最近的基站/机场候选点，不代表真实部署方案。
3. 默认频段、基站天线高度和安全余量是可调假设；当前没有真实发射功率、天线增益、接收灵敏度、噪声、干扰或设备协议参数。
4. 前端只使用 Cesium 场景当前已缓存的地形高度做演示级判断；采样不足时显示 `terrain pending`，不得补造遮挡结果。
5. Cesium World Terrain 仍不是工程 DEM/DSM。正式山体遮挡分析应接入本地 DEM/DSM 或受控高程服务，并输出可复现剖面采样记录。
6. 不允许从 Google 3D Tiles 提取、缓存或反向解析地形、建筑或高程数据。

## 十五、数据质量检查边界

当前新增结构化数据质量检查，输出：

1. `output/data_quality_report.json`
2. `data/processed/data_quality_issues.csv`
3. `/api/data-quality`
4. `/api/data-quality/issues`

质量检查覆盖：

1. 标准 ID 缺失或重复：`tower_id`、`route_id`、`waypoint_id`、`station_id`、`task_id`。
2. 坐标字段缺失、无法转为数字、超出泰安经验范围。
3. 经纬度反置风险。
4. KML 航线与最近杆塔匹配置信度和距离统计。

当前可验证事实：

1. 杆塔台账共 849 条记录，其中 831 条有有效经纬度，18 条缺失经纬度。
2. 831 条有效杆塔坐标均位于当前泰安经验范围内。
3. 当前未发现经纬度反置风险。
4. 当前 72 个 KML 航点均位于当前泰安经验范围内。
5. 当前 3 条 KML 航线的最近杆塔匹配置信度均为 `high`，但这仍是规则推断，不替代人工确认。

边界：

1. 泰安经验范围只是质量检查阈值，不是行政边界或工程边界。
2. 缺失坐标的杆塔不会被伪造位置，也不会出现在 GeoJSON 点图层中。
3. 匹配置信度来自空间距离和航线半径规则，不代表线路拓扑已确认。
4. 数据质量报告不输出真实安全风险、通信覆盖或高程结论。
## 十六、前端质量问题明细边界

前端右侧 `数据质量问题明细` 读取 `/api/data-quality/issues`，用于把后台质量检查结果转成可点击、可复核的验收对象。

边界：

1. 质量问题明细来自真实解析后的结构化检查结果，不是模拟值。
2. 当前 18 条问题是杆塔坐标缺失或无法转为数字；这些杆塔不会被伪造坐标，也不会出现在地图点图层。
3. 点击问题后显示的 `not_mappable_missing_coordinate` 只表示无法地图定位，不表示飞行风险等级。
4. 处理建议是数据修复建议，例如补齐台账坐标并重新运行数据管线；不输出真实 DEM/DSM、高程、通信覆盖或无人机能耗结论。

源记录定位字段：

1. `source_file` 和 `source_row` 来自标准化管线读取到的原始台账位置，用于回到 Excel 修复数据。
2. `line_name` 和 `tower_no` 用于人工识别杆塔对象。
3. `longitude` 和 `latitude` 在缺失坐标问题中保持为空，不填入默认值。
4. 前端展示这些字段仅用于数据整改闭环，不代表已经补齐真实坐标。

坐标补录模板边界：

1. `data/processed/coordinate_backfill_template.csv` 是由质量问题派生的整改辅助表。
2. `longitude_to_fill` 和 `latitude_to_fill` 必须由人工依据真实台账、现场资料或可信测绘数据补录。
3. 模板文件本身不是原始数据源；补齐后应回写原始台账，再重新运行标准化管线。
4. 模板不会自动补齐坐标，也不会用最近杆塔、航线点或地图底图推断坐标。

补录校验边界：

1. `data/processed/coordinate_backfill_validation.csv` 和 `output/coordinate_backfill_validation_report.json` 只检查补录模板中的待填字段。
2. 校验状态 `valid` 仅表示经纬度为数字且位于泰安经验范围内，不代表坐标已经通过现场或测绘复核。
3. `possible_lon_lat_swapped` 只表示数值范围疑似反置，需要人工确认。
4. 校验不会回写原始 Excel，也不会改变 `towers.csv`、GeoJSON 或地图点图层。

## 十七、未来数据导入接口边界

当前新增 `/api/import/*` 接口和 `scripts/import_data.ps1` 只用于把未来新增数据上传到 `data/raw/imports/<batch_id>/` 暂存区，并生成 `import_manifest.json`。该机制的边界如下：

1. 暂存导入不会覆盖现有原始数据。
2. 暂存导入不会自动进入 `data/processed/`，也不会自动刷新 SQLite、GeoJSON 或前端图层。
3. 暂存导入不会推断字段含义、坐标顺序、单位或文件与任务之间的对应关系。
4. 新的任务表、台账或 KML 必须人工确认后，放入对应 `data/raw/tasks/`、`data/raw/ledger/`、`data/raw/kml/` 目录，再重新运行标准化数据管线。
5. 当前另有 `/api/import/kml-routes` 和 `scripts/import_kml_routes.ps1` 可把 `.kml` 直接新增到 `data/raw/kml/` 并重建 processed 数据；该能力只表示 KML 已进入现有航线解析管线，不代表航线与任务、线路或杆塔关系已人工确认。
6. 新的 DEM/DSM 文件只是暂存；在本地地形采样管线实现前，不作为真实工程高程、AGL、剖面或通信遮挡计算源。
7. 新的通信参数和无人机参数只是暂存；在模型规则和验证数据明确前，不输出真实通信覆盖、最大通信半径、链路质量或能耗结论。
8. 导入 manifest 中的 SHA-256 只用于文件追踪和复核，不代表数据内容已经通过工程验收。
