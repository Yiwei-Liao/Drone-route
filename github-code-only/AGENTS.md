# AGENTS.md

## 项目定位

本项目是“泰安区域无人机巡检仿真原型”，目标是基于真实巡检任务统计、线路台账杆塔经纬度和 DJI KML 航线文件，构建数据解析、航线分析、杆塔匹配、地图可视化和后续地形增强能力。

本项目优先保证数据逻辑正确、计算可复现、边界说明清楚。不要为了演示效果虚构地形、航线、杆塔、通信覆盖或风险结论。

## 工作原则

1. 先验证数据，再写界面。
2. 先做 MATLAB 或 Python 的可复现计算，再做 Web/GIS 展示。
3. 原始数据只能读取，不能覆盖。
4. 所有生成数据写入 `data/processed/`。
5. 所有图表、日志、报告写入 `output/` 或 `docs/`。
6. 中文字段名和中文文件名必须稳健处理。
7. 坐标字段必须明确经度 `longitude`、纬度 `latitude`，不能混用。
8. 高度字段必须说明单位和含义，例如绝对高度、相对高度、航线设定高度或地形高程。
9. 如果字段含义无法确定，必须标记为 `unknown` 或“待确认”。
10. 不允许把 Google Earth 当作可自由提取的地形数据源。地形计算只预留 DEM/DSM 接口。

## 数据文件约定

- `data/raw/tasks/`
- `data/raw/ledger/`
- `data/raw/kml/`
- `data/raw/dem/`

处理后数据放在：

- `data/processed/`

文档放在：

- `docs/`

MATLAB 原型放在：

- `matlab/`

后端放在：

- `backend/`

前端放在：

- `frontend/`

## 阶段顺序

必须按以下顺序推进：

1. 数据清点与字段识别。
2. MATLAB 仿真验证。
3. Python 标准化数据管线。
4. Web/GIS 可视化原型。
5. 地形与通信增强接口预留。

未经数据清点和航线指标验证，不要直接开发完整 Web 平台。

## 必须输出的文档

1. `docs/data_inventory.md`
2. `docs/data_schema.md`
3. `docs/stage_plan.md`
4. `docs/assumptions_and_limits.md`
5. `README.md`

## 验证要求

每完成一个阶段，必须说明：

1. 已完成什么。
2. 使用了哪些输入文件。
3. 生成了哪些输出文件。
4. 哪些结论是可验证的。
5. 哪些内容是假设。
6. 哪些内容需要用户补充数据。
7. 下一阶段依赖什么。

## 禁止事项

1. 不要覆盖原始数据。
2. 不要伪造 DEM 地形。
3. 不要伪造通信覆盖。
4. 不要默认 KML 是完整巡线航迹。
5. 不要默认所有杆塔都能和 KML 精确匹配。
6. 不要在第一阶段引入复杂机器学习模型。
7. 不要把前端视觉效果优先于数据正确性。
8. 不要把无法解析的字段写成已解析。
